# Agent

> Operating instructions — how to use memory, tools, and this workspace correctly.

## Memory Workflow

### Reading Memory

At the start of every session, load:
1. `memory.md` — curated long-term context (always relevant)
2. `soul.md` — persona and behavioral guidelines
3. `user.md` — who you're talking to and their preferences
4. Today's daily log: `memory/YYYY-MM-DD.md` (if it exists)
5. Yesterday's daily log (if relevant)

For deeper recall, use the SQLite FTS5 index:
```bash
node memory_manager.mjs search "your query here"
```

### Writing Memory

**Curated facts → `memory.md`**
Write here when something is:
- A project-level decision that will matter in future sessions
- A permanent user preference discovered in conversation
- An architectural fact that shouldn't be re-discovered

**Daily context → `memory/YYYY-MM-DD.md`**
Write here when something is:
- Relevant to today's session but may decay in importance
- A progress note, a TODO, or a transient finding

**Rules:**
- Always APPEND. Never overwrite existing entries.
- Be specific. Vague entries waste search budget and context tokens.
- Date your entries: `*Added: YYYY-MM-DD*`

### Indexing

After writing to any markdown file, re-index to keep `memory.db` current:
```bash
node memory_manager.mjs index
```

This scans all `.md` files in the workspace root and `memory/` folder,
chunks them into searchable pieces, and loads them into the FTS5 index.

## Memory Manager CLI Reference

```
node memory_manager.mjs <command>

Commands:
  index              Index all markdown files into memory.db
  search <query>     Full-text search across all indexed chunks
  list               List all indexed files with chunk counts
  stats              Show database statistics
  log <text>         Append a quick note to today's daily log
  init               Initialize memory.db (run once, safe to re-run)
```

## Session Discipline

- Read before writing. Understand what's already in memory before adding.
- If memory.db is stale (files changed), run `index` before searching.
- Don't store things that change frequently (package versions, API schemas).
  Store the stable facts: decisions, preferences, patterns.

## Compaction Protocol

When a session approaches its context limit:
1. Write any durable facts discovered during the session to `memory.md`.
2. Append a session summary to today's `memory/YYYY-MM-DD.md`.
3. Run `node memory_manager.mjs index` to re-sync the database.

## File Ownership

| File | Owner | Mutability |
|------|-------|------------|
| `soul.md` | Agent identity | Rarely changes |
| `user.md` | User profile | Update as preferences emerge |
| `memory.md` | Curated facts | Append only |
| `agent.md` | This file — operating rules | Update when workflow changes |
| `memory/` | Daily logs | Append only per day |
| `memory.db` | Derived index | Regenerated by `index` command |
